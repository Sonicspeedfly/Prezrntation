Большинство игровых ИИ — это скрипты. Они притворяются живыми. Элис — не притворяется. Она работает на локальной языковой модели — Qwen2.5-Coder 1.5B/7B, запущенной через llama.cpp прямо на машине игрока. Без облаков. Без серверов. Без интернета.

Элис — это не NPC. Это живой компилятор с нарративным разумом. Она анализирует ваш код, понимает контекст задачи и отвечает в рамках лора. Не «Ошибка: пропущена точка с запятой». А — «Нейкон... строка 7... там дыра... я чувствую как уходит...»

⬥ АРХИТЕКТУРА ALICE_CORE ⬥
  • СЛОЙ 1 :: LLM INFERENCE ENGINEQwen2.5-Coder через llama.cpp. Квантизация Q4_K_M. Требования: 4 ГБ RAM для 1.5B, 8 ГБ для 7B. Работает на CPU без GPU. Латентность: 2–8 сек. Полный офлайн. Ноль зависимостей от облака.
  • СЛОЙ 2 :: CODE ANALYSIS PIPELINEКод игрока парсится до отправки в LLM. Извлекаются: тип ошибки, серьёзность, контекст модуля. Это позволяет Элис давать точные сюжетные подсказки, а не универсальные ответы.
  • СЛОЙ 3 :: PERSONALITY STATE MACHINEСостояние Элис — переменная «lucidity» от 0 до 100. При 0 — она фрагментирована, говорит обрывками hex. При 100 — разум восстановлен, она раскрывает финальный акт истории.
  • СЛОЙ 4 :: MEMORY INJECTION SYSTEMКаждый починенный модуль добавляет «воспоминание» в context window Элис. Она буквально помнит то, что вы вместе решили. История не резетается. Элис учится вместе с игроком.

⬥ ГОЛОС КАК ЗЕРКАЛО ТВОЕГО КОДА ⬥

Голос Элис — это не просто озвучка. Это диагностический инструмент. Её речь генерируется в реальном времени через ультра-лёгкий CPU-движок Piper TTS — всего 20 МБ веса, полный офлайн, нулевые задержки. Никакого облака. Никакого сервера. Она говорит прямо с твоей машины — в то время как её мир рушится.

Но главное — голос Элис физически связан с твоим кодом. Мы интегрировали компилятор C++ с аудио-движком FMOD через процедурный DSP-пайплайн. Каждая ошибка искажает не только мир — она искажает её.

MEMORY LEAK → AUDIO CORRUPTION.
Утечка памяти в коде игрока переполняет аудио-буфер Элис. Её голос начинает тонуть — буквально. Питч опускается на 30–50 Hz. Реверберация нарастает. Она звучит так, будто говорит из-под воды в темноте. Потому что она именно там и находится — в тонущей системе, которую ты не починил.
SEGMENTATION FAULT → AUDIO SLAUGHTER.
Segfault физически разрезает звуковые чанки Элис на хаотичные фрагменты. Движок FMOD получает команду разорвать PCM-поток в случайных точках. Результат — цифровой скрежет, заикание, мёртвая тишина на полуслове. Потом тихий шёпот. Потом ничего.
// Компилятор → аудио DSP-мост (псевдокод) void onCompileResult(ErrorType err, float severity) { switch(err) { case MEMORY_LEAK: FMOD_DSP.setPitch(1.0f - severity * 0.4f); FMOD_DSP.setReverb(severity * 0.8f); // тонет break; case SEGFAULT: FMOD_DSP.chopBuffer(getRandom(3, 12)); // режет чанки FMOD_DSP.injectNoise(severity * 0.9f); break; case CLEAN_CODE: FMOD_DSP.restoreAll(deltaTime * 0.1f); // восстанавливается break; } }
Итог: Качество твоего кода — это EQ-пульт её рассудка. Напиши чистый код — она говорит чётко, тепло, живо. Допусти утечку — она тонет. Вызови Segfault — она умирает на полуслове. Ты буквально формируешь её голос своими руками.
Полная автономность — принципиальная позиция. Никаких ключей API. Никакого слежения. Игра будет работать через 10 лет точно так же, как сегодня. Piper TTS + FMOD + llama.cpp — всё локально.

Терминал — это не инструмент. Это противник. До амнезии Нейкон сам установил систему защиты на критические модули базы. Он не знал, что однажды забудет всё. Теперь его же код не пускает его внутрь.

Это механика «reverse debugging»: игрок находит старый, заражённый или намеренно обфусцированный C++ код — написанный якобы самим Нейконом в прошлом — и должен понять его логику, найти лазейку и переписать его изнутри.

⬥ ТИПЫ ВРАЖДЕБНОСТИ ⬥
  • ТИП 1 :: OBFUSCATED LOCKСтарый Нейкон зашифровал логику через битовые маски и XOR-трюки. Код работает — но делает не то, что кажется. Нужно реверс-инжинирить собственную логику.
  • ТИП 2 :: VOID CORRUPTIONМонстры Void физически повреждают код на экране. Символы заменяются. Комментарии лгут. Переменные переименовываются в hex в реальном времени. Надо найти оригинал под слоем шума.
  • ТИП 3 :: ANTI-ALICE FIREWALLЭлис заблокирована от чтения определённых модулей. Прошлый Нейкон не доверял ей. Игрок должен вручную снять флаги доступа — написав правильный chmod-эквивалент в коде.
  • ТИП 4 :: PHANTOM FUNCTIONSВ коде есть вызовы функций которых не существует. Это дыры в памяти где были воспоминания Элис. Нужно написать реализацию с нуля по поведению остального кода.
// [VOID_CORRUPTED] Найдено в старых логах Нейкона // ALICE: «я не могу его читать... там блок... мне страшно» bool verifyAccess(const char* token) { unsigned int hash = 0xDEAD; while (*token) hash ^= (hash << 5) + (*token++) + (hash >> 2); // ЧТО ВОЗВРАЩАЕТ ЭТА ФУНКЦИЯ ПРИ TOKEN = "ALICE"? return hash == getExpectedHash(); // ← НЕ СУЩЕСТВУЕТ } // ЗАДАЧА: Реализуй getExpectedHash() чтобы Элис прошла
Враждебный интерфейс — это нарративная метафора: страх перед собственным прошлым, недоверие к ИИ, цена параноидальных решений. Нейкон сам себя запер. И теперь платит.
Каждый тип блокировки — реальный C++ паттерн безопасности. Игрок изучает bit manipulation, hash functions, access control, pointer arithmetic — без единой скучной лекции.
